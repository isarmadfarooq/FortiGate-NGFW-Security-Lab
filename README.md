## рҹ“Ұ Task 1 вҖ” FortiGate VM Deployment

### Step 1 вҖ” Download FortiGate VM

Navigate to the official Fortinet support portal and download the VMware-compatible FortiGate VM image.

**Download URL:** https://support.fortinet.com/support/#/downloads/vm

```
Select: FortiGate вҶ’ VM Images вҶ’ VMware ESXi/Workstation вҶ’ FortiGate-VM64.ovf
```

After downloading, extract the `.zip` archive. You should see:

```
FortiGate-VM64/
в”ңв”Җв”Җ FortiGate-VM64.ovf       вҶҗ OVF descriptor (import this)
в”ңв”Җв”Җ FortiGate-VM64.mf        вҶҗ Manifest (checksums)
в”ңв”Җв”Җ fortios.vmdk             вҶҗ Primary disk image
в””в”Җв”Җ datadrive.vmdk           вҶҗ Data disk
```

> рҹ’Ў **Tip:** Verify the SHA256 checksum of the downloaded archive against the value published on the Fortinet portal before proceeding.

---

### Step 2 вҖ” Configure Virtual Network Interfaces

Open VMware Workstation's **Virtual Network Editor** to create the required host-only network adapters.

**Path:** `Edit вҶ’ Virtual Network Editor` *(requires Administrator privileges)*

Or search **"Virtual Network Editor"** from the Windows Start Menu.

```
Default networks available:
  VMNet1  вҶ’  Host-Only
  VMNet8  вҶ’  NAT

Add the following networks (Host-Only type):
  VMNet2  вҶ’  192.168.50.0/24    (LAN)
  VMNet3  вҶ’  [your choice]      (Additional zone)
  VMNet11 вҶ’  [your choice]      (Additional zone)
  VMNet12 вҶ’  [your choice]      (Additional zone)
```

**Configuration rules:**
- вң… Set all new adapters to **Host-Only** mode
- вң… Assign a **unique, non-overlapping subnet** to each
- вң… Disable DHCP on adapters (FortiGate will manage addressing)
- вқҢ Do NOT use NAT type for interface zones (only for management)

---

### Step 3 вҖ” Deploy FortiGate VM in VMware

Import the OVF template into VMware Workstation:

```
VMware Workstation вҶ’ File вҶ’ Open (Ctrl+O)
  в””в”Җв”Җ Select: FortiGate-VM64.ovf
      в”ңв”Җв”Җ Assign VM Name:    FortiGate-Lab
      в””в”Җв”Җ Storage Path:      [your preferred directory]
```

After import completes, configure VM hardware settings:

**`Edit Virtual Machine Settings` вҶ’ Set the following:**

| Hardware Component | Value |
|-------------------|-------|
| Memory (RAM) | **2048 MB (2 GB)** |
| Processors | 1 vCPU |
| Hard Disk | 30 GB |
| Network Adapter 1 | VMNet8 (NAT) вҶ’ port1 Management |
| Network Adapter 2 | VMNet2 (Host-Only) вҶ’ port2 WAN |
| Network Adapter 3 | VMNet3 (Host-Only) вҶ’ port3 LAN |
| Network Adapter 4 | VMNet11 (Host-Only) вҶ’ port4 |
| Network Adapter 5 | VMNet11 (Host-Only) вҶ’ port5 |
| Network Adapter 6 | VMNet12 (Host-Only) вҶ’ port6 |

> вҡ пёҸ The OVF import may take **3вҖ“10 minutes** depending on disk speed. Do not interrupt the process.

### Step 4 вҖ” Configure Management Interface via CLI

Boot the FortiGate VM and log in through the VMware console:

```
Login:    admin
Password: [press Enter вҖ” blank by default]
          вҶ’ You will be prompted to set a new password immediately
```

**Verify current interface state:**

```bash
get system interface
```

**Configure port1 as the management interface:**

```bash
config system interface
    edit port1
        set mode static
        set ip 192.168.139.131 255.255.255.0
        set allowaccess http https ssh ping
    next
end
```

**Verify the configuration was applied:**

```bash
show system interface port1
```

Expected output excerpt:
```
config system interface
    edit "port1"
        set ip 192.168.139.131 255.255.255.0
        set allowaccess ping https ssh http
        set type physical
    next
end
```

> рҹ”җ **Security note:** Remove `http` and `telnet` from `allowaccess` in production. Use only `https` and `ssh`.

### Step 6 вҖ” Configure WAN and LAN Interfaces

Navigate to **Network вҶ’ Interfaces** in the GUI.

#### WAN Interface (port2)

```
Interface:        port2
Alias:            WAN
Role:             WAN
Addressing Mode:  Manual
IP/Netmask:       [ISP-provided public IP] / [subnet mask]
Admin Access:     HTTPS, PING (minimal)
```

#### LAN Interface (port3)

```
Interface:        port3
Alias:            LAN
Role:             LAN
Addressing Mode:  Manual
IP/Netmask:       192.168.100.1 / 255.255.255.0
DHCP Server:      вң… Enable
  - IP Range:     192.168.100.10 вҖ“ 192.168.100.200
  - DNS Server 1: 8.8.8.8
  - DNS Server 2: 8.8.4.4
```

#### Add Default Static Route

**Network вҶ’ Static Routes вҶ’ Create New**

```
Destination:  Subnet вҶ’ 0.0.0.0 / 0.0.0.0
Gateway:      [ISP gateway IP]
Interface:    port2
Distance:     10
```

> This 0.0.0.0/0 default route directs all internet-bound traffic through the WAN interface to the ISP gateway.

### Step 2 вҖ” Configure Admin Credentials

Navigate to **System вҶ’ Administrators вҶ’ admin вҶ’ Edit**

| Setting | Recommended Value |
|---------|-----------------|

| Password | Min. 12 chars: uppercase + lowercase + digits + symbols |

  в””в”Җв”Җ Use Outgoing Interface Address

### Policy C вҖ” Web Content Filtering

#### 1. Enable Web Filter Feature

**System вҶ’ Feature Visibility**

```
Web Filter:   вң… Toggle ON вҶ’ Apply
```

#### 2. Configure URL Filter Profile

**Security Profiles вҶ’ Web Filter вҶ’ [default] вҶ’ Static URL Filter вҶ’ URL Filter вҶ’ Create**

```
URL:     facebook.com
Type:    Wildcard          вҶҗ Matches *.facebook.com and all subdomains
Action:  Block
Status:  вң… Enable
```

Click **OK** to save.

> The **Wildcard** type ensures subdomains like `m.facebook.com`, `static.xx.fbcdn.net`, and `connect.facebook.net` are also blocked, preventing evasion.

#### 3. Create Web Filter Firewall Policy

**Policy & Objects вҶ’ Firewall Policy вҶ’ Create New**

```
Name:                         No-Facebook-Internet-Access
Incoming Interface:           port3 (LAN)
Outgoing Interface:           port2 (WAN)
Source:                       all
Destination:                  all
Service:                      ALL
Action:                       ACCEPT
NAT:                          вң… Enable вҖ” Use Outgoing Interface Address
Security Profiles:
  в””в”Җв”Җ Web Filter:             вң… [your web filter profile with facebook block]
```

#### 4. Set Policy Order вҡ пёҸ CRITICAL

**Policy & Objects вҶ’ Firewall Policy вҶ’ View By Sequence**

Drag `No-Facebook-Internet-Access` to **position #1** (above `Internet-Traffic`).

```
Policy Evaluation Order:
  [1] No-Facebook-Internet-Access   вҶҗ Web filter applied here FIRST
  [2] Internet-Traffic               вҶҗ General internet access
  [3] (implicit deny all)
```

> вқ— **If this policy is below `Internet-Traffic`, the web filter will never trigger.** FortiGate evaluates policies top-to-bottom and stops at the first match.

## рҹ”§ Troubleshooting

<details>
<summary><strong>вқ“ Cannot access FortiGate GUI after setting IP</strong></summary>

```bash
# Verify the IP was set correctly via CLI
show system interface port1

# Check that allowaccess includes https
# If not, re-run:
config system interface
    edit port1
        set allowaccess https ssh ping
    next
end

# Verify your host machine is on the same subnet
# Host IP should be in 192.168.139.0/24 range
```
</details>

<details>
<summary><strong>вқ“ Browser shows certificate error for FortiGate GUI</strong></summary>

This is expected. FortiGate uses a self-signed certificate by default.

- **Chrome:** Click **Advanced** вҶ’ **Proceed to 192.168.139.131 (unsafe)**
- **Firefox:** Click **Advanced** вҶ’ **Accept the Risk and Continue**
- For production: Install a valid certificate via **System вҶ’ Certificates**
</details>

<details>
<summary><strong>вқ“ LAN clients cannot reach the internet</strong></summary>

Check in order:
1. **Default route exists:** Network вҶ’ Static Routes вҶ’ verify 0.0.0.0/0 route
2. **IPv4 policy exists:** Policy & Objects вҶ’ Firewall Policy вҶ’ verify Internet-Traffic policy
3. **NAT is enabled** on the firewall policy
4. **WAN interface IP is correct** (matches ISP configuration)
5. **DNS is configured:** Network вҶ’ DNS вҶ’ set DNS servers (8.8.8.8, 8.8.4.4)
</details>

<details>
<summary><strong>вқ“ facebook.com is not being blocked</strong></summary>

1. Verify the web filter policy is **at the TOP** of the policy list (By Sequence view)
2. Confirm the web filter profile is **attached** to the policy (Security Profiles section)
3. Check that the URL entry is `facebook.com`, Type `Wildcard`, Action `Block`, Status `Enabled`
4. Clear browser cache and retry (browser may have cached the page)
5. Check FortiGate web filter logs: Log & Report вҶ’ Security Events вҶ’ Web Filter
</details>

<details>
<summary><strong>вқ“ DoS policy is not triggering</strong></summary>

1. Verify the DoS policy interface is set to the **WAN interface** (where attack traffic arrives)
2. Check that the specific anomaly profile (e.g., `tcp_syn_flood`) status is **Enabled**
3. Check that Action is set to **Block** (not Pass)
4. Verify the hping3 command is targeting the correct IP address
5. Check logs: Log & Report вҶ’ Security Events вҶ’ DoS
</details>

<details>
<summary><strong>вқ“ VMware shows import error for FortiGate OVF</strong></summary>

```
Try: File вҶ’ Open вҶ’ Browse to .ovf file (not .vmx)
If error persists, use VMware OVFTool:
  ovftool FortiGate-VM64.ovf output/FortiGate-VM64.vmx
```
</details>
